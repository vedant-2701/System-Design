# gRPC OrderService — Implementation

A production-structured gRPC service in Go demonstrating core gRPC patterns.

---

## Project Structure

```
grpc-demo/
├── proto/
│   └── order.proto          # Source of truth: schema + service definition
├── gen/
│   └── order/               # Generated by protoc — DO NOT EDIT
│       ├── order.pb.go      # Message types (Go structs)
│       └── order_grpc.pb.go # Server interface + client stub
├── interceptors/
│   └── interceptors.go      # Recovery, Logging, Auth middleware
├── server/
│   └── server.go            # OrderService implementation
├── client/
│   └── main.go              # Client exercising all RPC patterns
├── main.go                  # Server entry point, wiring, graceful shutdown
├── setup.sh                 # Build script
└── go.mod
```

---

## RPC Patterns Implemented

| RPC | Type | Pattern |
|-----|------|---------|
| `GetOrder` | Unary | One request → one response |
| `CreateOrder` | Unary | One request → one response + fan-out to subscribers |
| `WatchOrders` | Server Streaming | One request → infinite responses until client disconnects |

## Running Locally

### Prerequisites

```bash
# Install Go 1.21+
# https://go.dev/dl/

# Install protoc
# macOS:    brew install protobuf
# Ubuntu:   apt install protobuf-compiler
# Windows:  https://github.com/protocolbuffers/protobuf/releases

# Install Go protoc plugins
go install google.golang.org/protobuf/cmd/protoc-gen-go@latest

```

### Build and Run

```bash
chmod +x setup.sh
./setup.sh

# Terminal 1 — start server
./bin/server

# Terminal 2 — run client demo
./bin/client
```

### Why `UnimplementedOrderServiceServer` is embedded

```go
type OrderServer struct {
    pb.UnimplementedOrderServiceServer
    // ...
}
```

When a new RPC is added to the `.proto` and code is regenerated, any server
that doesn't implement the new method gets the default from
`UnimplementedOrderServiceServer` — which returns `UNIMPLEMENTED`. Without this
embedding, the server struct no longer satisfies the interface and **won't compile**.

This is the forward-compatibility mechanism for server implementations.

### Why fan-out in CreateOrder is non-blocking

```go
select {
case ch <- event:
default:
    // drop: subscriber is slow
}
```

A slow `WatchOrders` subscriber must not back-pressure `CreateOrder`. If we used
a blocking send, one slow streaming client could stall all order creation. The
non-blocking pattern means fast writers are never blocked by slow readers.

In production: track dropped events with a Prometheus counter and alert when the
drop rate is non-zero — it means subscriber buffers are too small or the
consumer is too slow.

### Why the interceptor order is Recovery → Logging → Auth

**Recovery outermost**: if `UnaryLogging` or `UnaryAuth` panics, `UnaryRecovery`
still catches it. If Recovery were innermost, a panic in Logging would propagate
unhandled.

**Logging before Auth**: unauthenticated requests are still logged. This is
critical for:
- Detecting brute-force attacks (auth failure spike)
- Forensics when a security incident occurs
- Rate limiting (which reads logs/metrics before rejecting)

**Auth before Handler**: ensures the handler never runs for unauthenticated callers.

### Why deadlines are set per-RPC at the client, not globally

```go
ctx, cancel := context.WithTimeout(baseCtx, 5*time.Second)
defer cancel()
resp, err := client.GetOrder(ctx, req)
```

Different RPCs have different latency budgets:
- `GetOrder` (cache lookup) → 100ms
- `CreateOrder` (write + fan-out) → 2s
- `WatchOrders` (streaming) → no timeout, or minutes

A global connection timeout can't express these distinctions. Per-RPC deadlines
propagate to the server via gRPC metadata — the server knows how much time
the client is willing to wait and can cancel downstream calls accordingly.

### proto3 default value problem with `order_id = 0`

In proto3, `uint32` defaults to `0` when not set. This creates an ambiguity:
did the client send `order_id: 0` intentionally, or did they forget to set it?

Solutions in production:
1. Use `optional uint32 order_id = 1` (proto3 optional field — wrapper type)
2. Use `string order_id` with a UUID instead of auto-increment integers
3. Validate at the handler level and return `INVALID_ARGUMENT` for `order_id == 0`

This demo uses option 3 for simplicity.

## gRPC Status Codes Used

| Code | Where | Why |
|------|-------|-----|
| `NOT_FOUND` | `GetOrder` | Order ID doesn't exist |
| `INVALID_ARGUMENT` | `GetOrder`, `CreateOrder` | Missing/invalid input |
| `UNAUTHENTICATED` | Auth interceptor | Missing or invalid token |
| `INTERNAL` | Recovery interceptor | Panic converted to safe error |

### Should I retry on these?

- `NOT_FOUND` → **No**. The resource doesn't exist; retrying won't help.
- `INVALID_ARGUMENT` → **No**. The request is broken; fix the client.
- `UNAUTHENTICATED` → **No**. Refresh the token first, then retry.
- `UNAVAILABLE` → **Yes**. Server temporarily down; retry with backoff.
- `DEADLINE_EXCEEDED` → **Maybe**. Retry only if the operation is idempotent.

## What's Missing (Production Concerns)

1. **TLS** — replace `insecure.NewCredentials()` with mTLS for service-to-service auth
2. **Real JWT validation** — replace stub `validateToken` with RS256 signature verification
3. **Database** — replace in-memory map with a real repository layer
4. **Metrics** — add Prometheus counters/histograms per RPC method via interceptor
5. **Tracing** — propagate OpenTelemetry trace context via metadata
6. **Rate limiting** — add a rate-limiter interceptor before Auth
7. **gRPC-Gateway** — expose REST/JSON interface from the same `.proto`
8. **Load balancing** — configure `dns:///` resolver + `round_robin` policy for K8s
