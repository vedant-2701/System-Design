# Question

> Your API is processing a payment. The client sends a POST request with a valid card and amount. Your service successfully charges the card. But then your database write fails — you can't record the transaction.
> What status code do you return to the client, and why is this decision genuinely difficult? What are the consequences of getting it wrong in either direction?

---

# Answer

You've identified the core tension correctly — this is genuinely hard. But your answer drifted toward a solution before fully answering the question I asked.

Let me push on the status code decision specifically, then we'll talk solutions.

**The brutal reality:**

If you return **500** — client sees failure. They retry. But the card was already charged. Now you've potentially double-charged them. The client has no idea the charge went through.

If you return **200** — client sees success. But your system has no record of the transaction. No receipt, no reconciliation, no audit trail. The money moved but your system is blind to it.

**Both are wrong. Neither is safe.** This is what makes it hard. It's not a technical question — it's a consistency question at the boundary of two systems you don't fully control.

Your instinct about idempotency is right but incomplete. Let me sharpen it.

---

The real answer engineers use in production:

**You should never be in this situation by design.**

Payment systems use the **Transactional Outbox Pattern** or **two-phase approach**:

1. Write the transaction record to DB *first*, in a pending state
2. Then charge the card
3. Then update the record to confirmed

If step 3 fails, you have a pending record. A background reconciliation job detects it and resolves it.

If step 2 fails, no money moved, pending record exists, you can safely retry or cancel.

**The database write should happen before the external side effect — not after.**

---

But you asked what status code to return *given* the failure already happened.

The honest answer: **return 500, but with an idempotency key system.**

The client retries with the same idempotency key. Your payment provider recognizes the key and returns the original charge result without double-charging. You attempt the DB write again. This is how Stripe handles it.
