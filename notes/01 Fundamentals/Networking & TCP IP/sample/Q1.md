# Question

> Your Go backend service running in a private subnet makes 1000 concurrent outbound HTTP calls to an external API. Each call is a new TCP connection. What problem could NAT cause here, and what's the theoretical limit?

---

# Answer

## Port Exhaustion Through NAT — Complete Picture

Your Go service has one private IP. The NAT gateway has one public IP. Every outbound TCP connection gets mapped to:

```
(public IP, public port) → (private IP, private port)
```

The NAT gateway is the one consuming public ports — not your service directly. And yes, ports are 16-bit = 65535 total, with ~64k ephemeral ports available (typically `1024–65535`, exact range is OS-configurable via `/proc/sys/net/ipv4/ip_local_port_range` on Linux).

So the hard limit is: **~64k simultaneous outbound connections from one public IP to one destination IP:port pair.**

But here's what you missed — the limit is actually **per (source IP, destination IP, destination port) tuple**, not globally per source IP.

A TCP connection is uniquely identified by:
```
(source IP, source port, destination IP, destination port)
```

So if you're calling two different external APIs:
- `api1.example.com:443` → 64k connections available
- `api2.example.com:443` → another 64k connections available

The exhaustion only bites when you have 64k+ *concurrent* connections to the **same destination IP:port**. For most services that's not the issue. The real production issue is different.

---

## The Real Production Problem — TIME_WAIT Accumulation

With 1000 concurrent short-lived HTTP connections, the issue isn't hitting 64k simultaneously. It's that **closed TCP connections don't immediately release their ports.**

When a TCP connection closes, the port enters `TIME_WAIT` state for `2 * MSL` (Maximum Segment Lifetime) — typically **60 seconds on Linux**.

So if your service is making 1000 new connections per second to the same destination, after 60 seconds you have 60,000 ports in TIME_WAIT — nearly exhausted. New connections start failing with `EADDRNOTAVAIL`.

```
1000 req/sec × 60 sec TIME_WAIT = 60,000 ports consumed
Available ephemeral ports ≈ 64,000
→ Nearly exhausted after 60 seconds
```

This is a real production incident pattern. Services look healthy, then start throwing connection errors under sustained load.

---

## How This Is Actually Solved in Production

**Connection pooling** — the real fix. Reuse TCP connections instead of creating new ones per request. `http.Client` in Go does this by default via `Transport`. A pool of 50 persistent connections handles thousands of requests/second with zero port exhaustion.

**SO_REUSEADDR / SO_REUSEPORT** — allows binding to ports still in TIME_WAIT.

**Multiple NAT IPs** — AWS NAT Gateway scales by adding more public IPs, multiplying your port budget.

**Tuning TIME_WAIT** — `tcp_tw_reuse` on Linux allows reusing TIME_WAIT sockets for new outbound connections. Not always safe but common in high-throughput services.
