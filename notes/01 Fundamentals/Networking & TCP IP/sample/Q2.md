# QUestion

> Your Go http.Client uses a connection pool with MaxIdleConnsPerHost = 10 (the default). Your service is making 500 requests/second to the same upstream. What happens to the connections that can't fit in the pool? And what does that mean for port exhaustion?

---

# Answer

## What Actually Happens With Pool Size 10

```
500 req/sec arrive
       ↓
Pool has 10 idle connections
       ↓
10 requests reuse pooled connections  ← no new TCP, no handshake
490 requests create NEW connections   ← full TCP (+ TLS) handshake each
       ↓
Requests complete
       ↓
10 connections return to pool
490 connections don't fit → closed immediately
       ↓
490 connections enter TIME_WAIT (60 sec)
```

After 60 seconds of steady traffic:
```
490 new connections/sec × 60 sec = 29,400 ports in TIME_WAIT
```

At 1000 req/sec with pool size 10:
```
990 × 60 = 59,400 ports in TIME_WAIT → nearly exhausted
```

One traffic spike tips you into `EADDRNOTAVAIL`. Service starts throwing errors. Looks like a mystery — your application code is fine, your upstream is fine. It's the port table.

---

## The Latency Cost You Didn't Mention

Port exhaustion is the catastrophic failure. But even before that, there's a silent performance cost.

Every connection outside the pool pays handshake latency:

```
TCP handshake:  1 RTT   (~1-2ms local, 50-200ms cross-region)
TLS handshake:  1-2 RTT additional
                ─────────────────
Total overhead: 2-3 RTT before first byte of HTTP data
```

At 490 extra connections/second, you're burning hundreds of milliseconds of unnecessary latency on every request that misses the pool. Under high load, this also means more concurrent connections in-flight, which increases memory pressure.

---

## The Fix Is Simple But Requires Understanding Why

```go
transport := &http.Transport{
    MaxIdleConnsPerHost: 100,  // match your concurrency
    MaxConnsPerHost:     200,  // hard cap on total connections
    IdleConnTimeout:     90 * time.Second,
}
client := &http.Client{Transport: transport}
```

`MaxIdleConnsPerHost` should be sized to your actual concurrency — not left at the default 10. Most production services get this wrong and silently pay the latency and port cost.

---

## The Full Mental Model

```
Pool size too small
    → excess connections created and immediately discarded
    → TIME_WAIT accumulation
    → port exhaustion under sustained load
    → EADDRNOTAVAIL errors

Pool size right-sized
    → connections reused
    → no TIME_WAIT accumulation
    → handshake cost paid once per connection lifecycle
    → stable under load
```
