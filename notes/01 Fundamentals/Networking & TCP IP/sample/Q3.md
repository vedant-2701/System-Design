# Question

> Go handles 500 req/sec concurrently. But `http.Client` in Go is backed by a connection pool, and the pool has `MaxConnsPerHost` as a hard ceiling. If that ceiling is hit and all connections are in use — what does Go do with the 501st request? Does it error immediately, block, or something else?

---

# Answer

Go's http.Transport blocks the goroutine making the request until a connection becomes available from the pool. This is the only sensible behavior for a connection pool — erroring immediately would make the client code complex (constant retries), and creating unbounded connections defeats the purpose of the ceiling.

But this introduces a new problem worth naming.

If your upstream is slow — say each request takes 500ms instead of 10ms — connections are held longer, the pool stays saturated, and the queue of blocked goroutines grows. Each blocked goroutine in Go consumes ~2-8KB of stack memory. At 10,000 blocked goroutines you're looking at 20-80MB just in goroutine stacks, plus the queue pressure.

This is how a slow upstream cascades into memory pressure on your service. You're not failing fast — you're accumulating waiters. This is exactly the problem the Circuit Breaker pattern solves, but that's a distributed systems topic for later.
The fix here is context.WithTimeout on every outbound call — so blocked goroutines don't wait forever.
