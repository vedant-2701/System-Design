# Question

> Your service handles video uploads. Users upload 500MB files. You notice that upload throughput starts low and increases over time before plateauing.
> Someone on your team suggests: "Let's just increase the initial cwnd on our servers to fix the slow start problem."
>
> Two questions:
>
> * Is this actually possible to configure, and would it help?
> * What does it NOT fix, and what's the remaining bottleneck after slow start ramps up?

---

# Answer

## Q1 — Yes, Initial cwnd Is Configurable

On Linux:

```bash
ip route change default via <gateway> initcwnd 10
```

Google actually pushed for this. In 2010 they published research showing that increasing initial cwnd from 3 to 10 MSS reduced page load times by 10%. Linux kernel default was changed to 10 MSS as a result. This is now standard practice.

What it means concretely:

```
Default initcwnd=1:   starts at ~1.4KB in-flight
initcwnd=10:          starts at ~14KB in-flight
```

For a 500MB upload, this helps the ramp-up phase start faster. But your instinct in Q2 is exactly right — let's make it precise.

---

## Q2 — What It Doesn't Fix

Increasing initcwnd just moves the starting line. The slow start curve still exists — it's just shifted up:

```
cwnd
 ^
 |              /—————— plateau (bottleneck)
 |             /
 |            /   ← with initcwnd=10
 |           /
 |      /———
 |     /      ← with initcwnd=1
 |    /
 └──────────────────────→ time
```

The shape of the curve is identical. You skip the first few doublings, saving a few RTTs. For a **500MB upload**, slow start ramps up quickly and is amortized over the entire transfer — the slow start penalty is maybe 200-500ms on a multi-second or multi-minute upload. Not your real problem.

---

## So What Is The Real Bottleneck?

After slow start completes and cwnd is fully ramped, throughput plateaus at:

```
throughput ≈ cwnd / RTT
```

The actual ceiling is determined by whichever of these is smallest:

```
1. Receiver window (rwnd)     — receiver's buffer size
2. Congestion window (cwnd)   — network capacity estimate
3. Available bandwidth        — physical link capacity
4. RTT                        — latency between client and server
```

For a 500MB upload from a client to your server, the dominant factors are usually:

**Client's upload bandwidth** — most asymmetric internet connections have far less upload than download. A client with 10 Mbps upload takes 400 seconds to upload 500MB regardless of TCP tuning.

**RTT** — throughput is fundamentally bounded by:
```
max throughput = window_size / RTT
```

If RTT is 100ms and cwnd plateaus at 1MB:
```
max throughput = 1MB / 0.1s = 10 MB/s = 80 Mbps
```

Increasing cwnd helps here — but only until you hit the actual link bandwidth.

**Packet loss rate** — this is the brutal one. There's a formula called the **Mathis equation**:

```
throughput ≈ MSS / (RTT × √loss_rate)
```

At 1% packet loss, RTT 100ms:
```
throughput ≈ 1460 / (0.1 × √0.01) = 1460 / 0.01 = 146,000 bytes/sec ≈ 1.1 Mbps
```

1% packet loss caps you at ~1 Mbps regardless of your bandwidth or cwnd. This is why TCP performs poorly on lossy networks — every loss event halves cwnd via AIMD. This is also why **QUIC (HTTP/3) uses UDP** — it implements its own loss recovery that doesn't punish the entire connection for a single lost packet.

## The Complete Answer

```
initcwnd increase helps:     short connections, high RTT paths
                             (web pages, API calls, stream start)

initcwnd increase doesn't fix:
  - client bandwidth ceiling
  - high RTT (throughput = window/RTT, RTT unchanged)
  - packet loss (AIMD still halves cwnd on every loss)
  - receiver buffer limits
```

For your 500MB upload specifically, the right solutions are:

**Chunked parallel uploads** — split the file, upload chunks concurrently over multiple connections. Each connection has its own cwnd, and you're utilizing bandwidth in parallel. S3 multipart upload works exactly this way.

**Upload closer to the user** — reduce RTT by accepting uploads at an edge location (like CloudFront or a regional endpoint), then transferring internally over a high-bandwidth low-latency backbone.

**Application-level flow control** — pre-check client bandwidth before starting, adapt chunk size dynamically.
